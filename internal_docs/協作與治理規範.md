# 立國專案｜協作與治理規範

- **版本：** v1.0
- **日期：** 2026-02-26
- **專案：** 立國工程 MCR／出缺勤系統（LeeKuo Repo）

本文件定義專案**角色分工**、**產物歸屬**、**分支與合併規則**及**交付發布流程**，供需求方（立國）與開發方（矽聯科技）共同遵守。

---

## 1. 角色與職責

| 角色 | 負責方 | 職責範圍 |
| --- | --- | --- |
| **需求與業務** | 立國專案負責人、專案團隊 | 訪談、需求整理、PRD 撰寫、雛型設計與定稿、使用者測試與導入、上線後變更需求提出 |
| **開發與迭代** | 矽聯科技顧問暨 PM、開發團隊 | 依 PRD 與雛型實作、單元／整合測試、程式迭代、配合變更修改與發布 |

**交付節奏**：需求方產出 PRD／雛型 → 開發方開發與測試迭代 → 交付 → 需求方與使用者測試導入 → 若有變更，雙方協同一次更改後發布。

### 1.1 與工程師之協作協議（現行）

- **需求方（立國）**：分享本 GitHub 專案（需求、PRD、雛型、文件）予工程方參考；授權本 Repo 給負責開發之人員，供其讀取。
- **開發方（矽聯）**：依需求方提供之資料，在約定之工程／開發環境內進行專案管理與實作；每週與需求方對齊進度與待辦。
- **每週同步**：需求方與開發團隊**每週**定期討論**開發進度**與**程式測試修正**，對齊待決議與下一階段目標。
- **客戶回饋**：當有回饋或需求調整時，需求方通知開發方；開發方進行變更與差異分析；需求方 PRD／雛型更新仍以本 repo 為準。
- **文件版本**：文件之版本與發布由需求方與專案經理共同管理；對外發布、客戶交付之文件與版次以本 repo 為依據。

## 2. 產物歸屬與工作邊界

### 2.1 需求方主責（建議由需求方提交或審核後合併）

- **01_Discovery/**：訪談紀錄、客戶資料卡、需求展開
- **02_Analysis/**：分析文件、需求整理、情境彙整、流程圖
- **03_Solution/**：PRD、規格草案、移交摘要、表單對照、簡報、專案進行進度
- **docs/**：專案總覽、對外文件（MkDocs 來源）、操作手冊與 SOP、待決議事項、任務與甘特圖
- **internal_docs/**：協作與治理規範、文件治理、程式編碼原則、本專案目錄對應說明
- **00_Templates/**：範本庫、Markdown 合規規範、立國專案專案管理與文件治理規則
- **99_Trace/**：已歸檔之需求、情境、版本紀錄

上述內容若有變更，建議由需求方 commit 或由需求方 Approve PR 後再合併。

### 2.2 開發方主責（建議由開發方提交，需求方可審核）

- **03_Solution/** 內之技術規格細化、實作對照（若專案採用）
- **程式碼、設定檔、CI/CD**：實作、測試、部署腳本（若程式庫置於本 Repo 或約定目錄）
- **.github/workflows/**：可依共識由開發方維護，重大變更建議經需求方知悉

### 2.3 共同遵守

- **.cursorrules**、**mkdocs.yml**、**根目錄說明檔**：重大變更建議雙方溝通後再改。
- **命名與格式**：遵守 `.cursorrules` 與 [程式編碼原則｜共用規範](程式編碼原則_共用規範.md)（程式代號、雛型檔名等）。

### 2.4 各功能作業 PRD 產出要求

提供各功能作業之 PRD 或新文件需求時，須依下列要求產出，作為工程方**框架、TABLE、API 命名規劃**之基礎，並與客戶對齊表單與報表名稱：

| 項目 | 說明 |
| --- | --- |
| **模組與主程式代碼** | 先建立完整之模組代號與主程式代號清單（權限表用）；子程式、批次、API／TABLE 命名見 [程式編碼原則｜共用規範](程式編碼原則_共用規範.md) §4～§5。 |
| **程式 × 使用者角色權限矩陣表** | 依情境產出：主程式代號 × 使用者角色之權限矩陣表（可執行／可查詢／不可見等）。 |
| **對應情境說明** | 各情境（如：現場回報、主管審核、請款關帳、查詢歷史等）之操作者角色、可存取程式、與流程說明。 |
| **模組欄位名稱中英文對照表** | 該模組所有客戶作業表單與報表之欄位：中文、英文、縮寫、用途表列；與客戶對齊表單與報表名稱；新欄位時提醒新增／對應／修改／刪除。見 [文件治理規範](docs_治理規範.md) §6。 |

上述 PRD 內容為框架設計、資料表（TABLE）命名、API 命名與權限設計之依據。

### 2.5 開發要求（含資安基本提醒）

開發方於規劃與實作時，除依 PRD 與雛型實作外，須注意下列資安基本要求：

| 類別 | 提醒要點 |
| --- | --- |
| **身分認證與登入** | 密碼不可明文儲存；登入／Session 逾時與登出機制。 |
| **授權與權限** | 依 PRD 之權限矩陣表與情境實作，採最小權限原則；前後端皆須驗證身分與權限。 |
| **敏感資料** | 個資、客戶資料之存取與儲存須受控；畫面與報表依角色遮罩或限制顯示。 |
| **傳輸** | 對外或敏感連線使用 HTTPS；API 與前後端通訊依環境要求加密。 |
| **日誌與稽核** | 重要操作可追溯；日誌內容避免記錄敏感資料。 |

## 3. 分支與合併規則

### 3.1 分支約定

- **main**：正式版；對外發布與使用者測試導入皆以 main 為準。
- **feature/xxx**、**fix/xxx**：功能或修正由開發方自 branch 開發，完成後以 Pull Request 合併至 main。

### 3.2 合併與審核

- **main 保護**：建議在 GitHub 設定 Branch protection rule，main 僅能透過 Pull Request 合併。
- **誰來審核**：涉及 01_Discovery、02_Analysis、03_Solution、docs、雛型之 PR，建議由需求方（或專案負責人）審核後合併；純程式／設定之 PR 可由開發方合併或依約定審核。

### 3.3 需求方之 PRD／雛型更新

- 需求方變更（PRD、雛型、docs）建議開 `docs/xxx` 或 `update/xxx` 分支，完成後開 PR，經審核後合併。

## 4. 交付與發布流程

### 4.1 開發交付

1. 開發方完成當輪實作與測試後，以 PR 合併至 main。
2. 需求方審核 PR（尤其涉及需求／雛型對齊），合併後即視為當輪交付。

### 4.2 使用者測試導入

1. 需求方以 main 為基準，與使用者進行測試與導入（文件、雛型或系統依專案階段）。
2. 發布管道依現有設定（如 MkDocs、GitHub Pages）從 main 建置／部署。

### 4.3 變更後協同發布

1. 使用者測試後若有變更需求，由需求方整理（必要時更新 PRD／雛型或待決議事項）。
2. 開發方依變更內容在 feature/fix branch 修改，完成後開 PR 至 main。
3. 需求方審核 PR，合併後觸發發布；發布完成後與使用者確認上線／導入結果。

- [ ] 雛型與文件命名遵守 `.cursorrules` 與程式代號規範。
- [ ] 重要里程碑或交付點可在 main 打 Tag，方便對應版本與發布。
