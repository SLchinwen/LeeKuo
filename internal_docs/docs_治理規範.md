# docs 文件治理規範｜立國專案

- **版本：** v1.0
- **日期：** 2026-02-26

本規範區分**對外（使用者／客戶）**與**內部（開發團隊）**文件，使產出與歸檔時能正確放置與撰寫。產生或新增文件時請依下列規則判斷歸屬。

---

## 1. 文件分類與目錄

| 對象 | 目錄 | 導覽區塊 | 用途 |
| --- | --- | --- | --- |
| **使用者／客戶** | `docs/` | MkDocs 全站導覽（首頁、專案管理、操作手冊與 SOP、待決議、任務與甘特圖等） | 專案總覽、操作手冊與 SOP、待決議事項、任務與甘特圖、模組說明與連結 |
| **開發團隊** | `internal_docs/` | 不透過 MkDocs 發布 | 協作與治理、文件治理、程式編碼原則、目錄對應說明 |

**判斷原則**：內容是否會給**客戶或 end-user** 看？若是 → 對外（`docs/`）；若為專案管理內部、設計、開發流程、內部規範 → 內部（`internal_docs/`）。內部文件不納入 MkDocs 建置，不對外發布。

---

## 2. 對外文件（使用者）規則

**放置位置**：`docs/index.md`（首頁）、`docs/立國專案管理_服務總覽.md`、`docs/操作手冊與SOP.md`、`docs/待決議事項.md`、`docs/立國專案_工作任務與甘特圖.md`，以及 `docs/MCR/`、`docs/HR/` 下之操作手冊與 SOP 等。

**禁止出現**：

- 設計或分析文件之**實體路徑**（如 `02_Analysis/…`、`internal_docs/…`）
- 內部**狀態**（如 Draft、待規劃、開發中）除非已約定對外呈現方式
- 版本控制、分支、PR、審核流程等**開發流程**細節
- 檔案命名規則、儲存位置等**實作細節**

**可出現**：

- 專案**目標**、**作業模式**、**程式清單**（程式代號、程式名稱、用途、雛型連結）
- **雛型**：僅以**連結**呈現，不寫實體路徑
- 文件**版次**與**發布／更新日**
- 操作手冊、SOP、教育訓練說明

### 2.1 操作手冊之對應畫面（對外雛型規則）

- **以雛型取代靜態圖片**：操作手冊中程式之「對應畫面」應以**最新版雛型**放置於 `docs/prototype/` 並以**連結**呈現作為操作說明，取代靜態截圖或外連圖片。
- **雛型放置**：對外可連結之雛型檔案（HTML、必要之匯入範本等）須同步放置於 `docs/prototype/`，與操作手冊內連結路徑對應，使 MkDocs 建置後使用者可直接開啟。
- **雛型內容**：雛型須含程式說明與操作步驟（可內建於雛型「📖 操作說明」彈窗，或另附操作說明文件）；更新雛型時須同步更新 `docs/prototype/` 內副本，以維持對外文件與最新版一致。

---

## 3. 內部文件（開發團隊）規則

**放置位置**：`internal_docs/` 下所有 Markdown 與子目錄。

**可出現**：

- 設計文件、PRD、規格之**路徑**與**索引**
- 文件或專案**狀態**（Draft、已發布、待規劃）
- **程式編碼原則**、雛型命名與發布路徑
- **協作與治理**、分支、PR、提交步驟
- **專案管理**文件與紀錄（會議、決策、變更歷程）
- 產出指引、撰寫範本

**歸檔**：專案管理或開發過程產生的紀錄，應放在 `internal_docs/` 或約定子目錄，並在 `internal_docs/index.md` 列出。

---

## 4. 對外網站（MkDocs）與內部文件

- **對外網站**：MkDocs 建置 **docs/** 全站，依 `mkdocs.yml` 之 `nav` 呈現首頁、專案管理、操作手冊與 SOP、待決議、任務與甘特圖等；導覽標題建議 ≤8 個中文字。
- **內部文件**：僅存於 repo 之 `internal_docs/`，供團隊在 Cursor／Git 中查閱；不透過 MkDocs 發布。

新增對外頁面時：更新 `docs/` 對應檔案並於 `mkdocs.yml` 之 `nav` 加入；新增內部文件時：置於 `internal_docs/`，並於 `internal_docs/index.md` 加入連結。

## 5. 產出時自檢

| 情境 | 應放置 | 內容注意 |
| --- | --- | --- |
| 給客戶看的總覽、程式清單、雛型入口 | `docs/` | 無內部路徑、雛型僅連結；更新日依系統日期填寫 |
| 操作手冊、SOP、教育訓練教材 | `docs/` 或 docs 下模組子目錄（如 MCR、HR） | 依 [立國專案專案管理與文件治理規則](../00_Templates/立國專案_專案管理與文件治理規則.md) 分階與命名 |
| 程式代號、編碼原則、雛型命名規範 | `internal_docs/` | 可有路徑與範例 |
| 協作、分支、PR、提交步驟 | `internal_docs/` | — |
| 專案管理、會議紀錄、變更紀錄 | `internal_docs/` 或 docs（若為對外總覽） | 對外僅職務稱謂，不寫個人姓名 |
| 設計文件、PRD、規格本體 | `02_Analysis/` 或 `03_Solution/` | 本專案 PRD、規格草案、移交摘要置於 03_Solution；分析產出置於 02_Analysis |
| **功能作業 PRD**（含權限矩陣、情境說明、欄位對照表） | `03_Solution/`（本專案慣例）或 02_Analysis | 須符合 **§6 標準分析產出要求**（四項必備） |

## 6. 標準分析產出要求（e 化專案）

本專案為 e 化專案，**各功能作業之 PRD 與分析產出**須符合下列**標準分析產出要求**，作為工程方**框架設計、TABLE 命名、API 命名與權限設計**之依據。詳見 [協作與治理規範](協作與治理規範.md) §2.4。

### 6.1 產出項目（四項必備）

| 項目 | 說明 |
| --- | --- |
| **模組與主程式代碼** | 該功能作業所屬模組及完整主程式代號清單（權限表用）；子程式、批次、API／TABLE 命名依 [程式編碼原則｜共用規範](程式編碼原則_共用規範.md) §4～§5。 |
| **程式 × 使用者角色權限矩陣表** | 主程式代號 × 使用者角色之權限矩陣（可執行／可查詢／不可見等）。 |
| **對應情境說明** | 各使用情境之操作者角色、可存取程式、流程與驗收說明。 |
| **模組欄位名稱中英文對照表** | 該模組所有客戶作業表單與報表之欄位：中文、英文、縮寫、用途表列；與客戶對齊表單與報表名稱。 |

### 6.2 放置與歸檔（本專案）

- 功能作業 PRD、規格草案、移交摘要：`03_Solution/`
- 權限矩陣表：可獨立成檔或併入 PRD 章節
- 欄位名稱對照表：本專案已建立 `03_Solution/表單與名詞對照清單_欄位引用標準.md`，新表單或欄位時更新同一檔或依專案約定新增

## 7. 雛型設計與產出（含資安建議）

雛型之**命名與放置**依 [程式編碼原則｜共用規範](程式編碼原則_共用規範.md)（雛型 HTML 檔名建議：`YYYY-MM-DD_prototype_<程式代號>.html`）；對外呈現僅以連結呈現，不寫實體路徑。

**對外操作手冊與雛型同步**：產出或更新雛型後，須將**最新版雛型**（含依賴之匯入範本等）複製至 `docs/prototype/`，並於操作手冊中以**雛型連結**作為該程式之「對應畫面／操作說明」，取代靜態圖片。詳見 §2.1。

### 7.1 雛型產出之一般要求

- **操作一致性**：同一專案內雛型之操作方式須一致（按鈕位置與樣式、表單送出／返回、選單展開與收合等）。
- **欄位說明之一致性**：同一欄位在雛型、PRD、欄位對照表中須一致。
- **資安要求考量**：權限區分示範、敏感欄位遮罩或假資料、登出／逾時示意、HTTPS 與後端驗證註記。

### 7.2 雛型設計時之資安建議

- **授權與權限**：依權限矩陣表在雛型中區分角色示範。
- **敏感資料**：雛型中敏感欄位以遮罩或假資料示範，不使用真實個資。
- **登入與登出**：保留登出按鈕或連結之示意。
- **傳輸與環境**：註明正式環境須以 HTTPS 提供；表單送出須經後端驗證。
