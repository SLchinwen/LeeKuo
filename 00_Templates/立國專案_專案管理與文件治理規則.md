# 立國專案｜專案管理與文件治理規則

- **版本：** v1.0
- **日期：** 2026-01-30
- **用途：** 建立專案管理手法與文件編排之規則與治理方式，供未來生成或新增文件時依循；缺資料時得依本規則互動補足。
- **適用對象：** 專案經理、顧問、開發團隊及所有產出或維護立國專案文件之人員與協作工具（含 AI 輔助產出）。

---

## 壹、專案管理手法

### 1.1 角色與分工

- **一律以職務稱謂呈現**，不對外揭露個人姓名（資安與個資考量）。
- **標準職務稱謂：** 顧問暨 PM、開發團隊、專案負責人、專案團隊；所屬標示為矽聯科技或立國。
- 協作方式：需求與待決由矽聯顧問暨 PM 與立國專案負責人／專案團隊對齊；決策列入待決議事項追蹤。

### 1.2 專案階段與會議

- **需求確認階段：** 以每周一次會議進行討論與決策；工作項須列出預估工作天，時程依工作天推算。
- **雛型確認後** 始進入 **系統開發階段**；系統開發時程依 **開發預估工時（工作天）** 推算，不得以猜測日期填寫。
- **假期假設：** 台灣農曆年春節（例：2026/02/14～02/23）不計入工作天；甘特圖須標示或排除。

### 1.3 甘特圖規範

- **分為兩大區塊：** 需求確認、系統開發。
- **需求確認：** 工作項（如需求收集、討論與決策、展開、分析、雛型製作、雛型確認）須填寫預估工作天，由工作天推算日期。
- **系統開發：** 各任務須填寫預估工作天，由專案經理或開發團隊填入，甘特圖依此推算；可註明日曆天與工作天之換算假設。
- **禁止：** 未經工作項展開與工作天數填寫而逕行猜測或隨意填寫甘特日期。

### 1.4 待決議事項

- **獨立頁面管理**，不與總覽混合長篇表格。
- **必備欄位：** 序、待決議內容、選項／備註、狀態、**決議日期**、**決議摘要**。
- 決策完成後須更新：狀態改為「已決」、決議日期填寫決議日、決議摘要簡述結論。

### 1.5 操作手冊與 SOP

- **已完成的系統** 其操作手冊與建議 SOP **發布於獨立頁面**（操作手冊與 SOP），不僅以「由矽聯提供連結」帶過。
- 新完成之模組上線後，須將操作步驟與 SOP 重點補入該獨立頁面或連結至該頁。

### 1.6 用語與呈現

- **對外文件（含 docs）：** 採用正式文件用詞，減少口語、對話式用詞。
- **專案管理文件：** 不放置技術規格與程式碼；技術文件由專案團隊另行維護。

## 貳、文件編排方式

### 2.1 目錄結構與用途

| 目錄 | 用途 | 是否對外（MkDocs） |
| --- | --- | --- |
| **docs/** | 專案管理、操作手冊與 SOP、待決議事項、任務與甘特圖等**客戶與 PM 檢視用**頁面 | 是（MkDocs 來源） |
| **03_Solution/** | PRD、規格草案、移交摘要、表單對照、流程圖、簡報、專案進行進度等**處方期產物** | 否（必要時可於 docs 連結或摘錄） |
| **02_Analysis/** | 需求整理、情境彙整、流程圖、變更對照、分析建議等**分析期產物** | 否 |
| **01_Discovery/** | 訪談、背景、診斷期產物 | 否 |
| **99_Trace/** | 已歸檔之需求、情境、版本紀錄等**追溯用** | 否 |
| **00_Templates/** | 範本、合規規範、**本治理規則** | 否（規則與範本引用用） |

### 2.2 命名規範

- **對外頁面（docs 內）：** 檔名可含專案或主題，例：`立國專案管理_服務總覽.md`、`操作手冊與SOP.md`、`待決議事項.md`。
- **導覽標題（mkdocs.yml nav）：** **不得超過 8 個中文字**；例：專案管理、服務總覽、任務與甘特圖、操作手冊與SOP、待決議事項。
- **一般文件：** 建議「主題_子類」或「模組_說明」，避免僅用英文縮寫造成混淆。

### 2.3 MkDocs 導覽結構（既定模式）

- **首頁** → index.md。
- **專案管理**（群組）→ 服務總覽、任務與甘特圖。
- **操作手冊與 SOP** → 獨立項次，不放在專案管理底下。
- **待決議事項** → 獨立項次。
- 新增對外頁面時，須同步更新 `mkdocs.yml` 之 `nav` 及必要時更新 `docs/index.md` 之連結。

### 2.4 表格與格式

- **Markdown 表格：** 須符合 `00_Templates/Markdown_合規規範.md`（含 MD060：分隔列為 `| --- | --- |`，管線右側有空格）。
- **連結：** 裸網址須以角括號包覆或改為 Markdown 連結，以通過 MD034。

### 2.5 文件頭部（建議）

- 對外或重要文件建議含：**日期、版本、用途、維護或更新日**；必要時註明「對應規格」或「來源文件」。

## 參、新增文件時之檢查清單

產出或新增「立國專案」相關文件時，請依下列項目檢核並依模式加入。

- [ ] **放置位置：** 屬對外／客戶檢視用 → `docs/`；屬規格／PRD／分析／移交 → `03_Solution/` 或 `02_Analysis/`；屬追溯 → `99_Trace/`。
- [ ] **命名：** 符合 2.2；若為 docs 且需出現在導覽，導覽標題 ≤8 字。
- [ ] **用語：** 正式文件用詞；人員以職務稱謂呈現，不寫姓名。
- [ ] **格式：** 遵守 `00_Templates/Markdown_合規規範.md`（清單、標題、表格、空白）。
- [ ] **總覽連結：** 若為新增之對外頁面，是否需在「專案管理服務總覽」之文件與連結整理加上連結？是否需在總覽內文（如程式清單、待決議）改為連結至新頁？
- [ ] **待決議事項：** 若新增或異動待決議項目，是否已更新 `docs/待決議事項.md` 並含決議日期／決議摘要欄位？
- [ ] **操作手冊與 SOP：** 若為已完成模組之操作手冊或 SOP，是否已納入或連結至 `docs/操作手冊與SOP.md`？
- [ ] **甘特圖：** 若涉及時程，是否以工作項與預估工作天推算，並區分需求確認與系統開發兩大塊？是否標示或排除約定假期？
- [ ] **MkDocs 導覽：** 若為 docs 新頁，是否已於 `mkdocs.yml` 之 `nav` 新增項目？是否已於 `docs/index.md` 新增對應區塊或連結？

## 肆、缺資料時之互動補足

生成或維護文件時，若資訊不足，應**主動標示需補足項目**並以互動方式請求補足，不得逕行猜測或留空不說明。

### 4.1 常見需補足項目

| 類型 | 需補足內容 | 建議詢問方式 |
| --- | --- | --- |
| 時程 | 預估工作天、開始日、假期假設 | 「請提供［任務名稱］之預估工作天（或開始日），以利甘特圖推算。」 |
| 決議 | 決議日期、決議摘要 | 「請提供第 N 項待決議之決議日期與決議摘要，以更新待決議事項頁。」 |
| 角色 | 職務與所屬（不要求姓名） | 「請確認對外文件中使用之職務稱謂（如專案負責人、顧問暨 PM）是否與現行一致。」 |
| 模組狀態 | 進行中／暫停／已完成 | 「請確認［模組名稱］目前狀態（進行中／暫停／已完成），以更新總覽與甘特圖。」 |
| 交付物 | 操作手冊、SOP 是否發布、連結 | 「［模組］是否已完成操作手冊或 SOP？若已完成，請確認是否發布於操作手冊與 SOP 頁或提供連結。」 |
| 導覽 | 新頁是否加入導覽、標題字數 | 「新增之［頁面名稱］是否加入 MkDocs 導覽？導覽標題建議 ≤8 字，請確認標題文字。」 |

### 4.2 文件內標示方式（供後續補足）

- 若某段內容待補，可於該段註記：**「【待補】請提供：……」** 或 **「【待確認】……」**。
- 表格欄位若暫無資料，可填「—」或「待補」，並於表下或章節說明「請於［會議／郵件］後補齊決議日期與摘要」。

### 4.3 引用本規則

- 產出或審閱立國專案相關文件時，應**引用並依循**本規則（`00_Templates/立國專案_專案管理與文件治理規則.md`）。
- 若本規則與專案實際約定衝突，以專案經理或客戶端書面確認為準；衝突解決後建議更新本規則版本與日期。

## 伍、規則修訂與版本

- 本規則之修訂由專案經理或顧問提案，經與客戶端對齊後更新版本與日期。
- 重大變更（如新增目錄、變更導覽結構、新增必備欄位）應於本文件中留下簡要紀錄，並同步更新相關範本或 .cursorrules 之引用說明。
