# 第二階段_需求整理_總覽
- **日期：** 2026-01-29
- **用途：** 以「**需求 → 情境 → 流程 → 需求展開**」順序整理第二階段文件，作為後續變更管理與工程移交基底。
- **來源文件：** `_Inbox` 第二階段需求相關文件

## 1. 需求（Why / What）
### 1.1 產品目標
- **回報與拍照拆線**：縮短回報時效
- **降低審核往返**：支援退回與補件
- **請款與文件依回報**：確保可追溯
- **吊車決策可用資料**：噸數與停駐點可用

### 1.2 成功指標（KPI）
- 回報完成時間下降 ≥ 30%
- 審核退回次數下降
- 因照片不齊導致回報卡關趨近 0
- 請款補件次數下降

## 2. 情境（Scenario）
### 2.1 主線情境
- SC-03 現場完工回報（回報×拍照雙軌）
- SC-03A 設備完工回報（通知可請款，可重開修改）
- SC-04 回報後補件與圖片換圖
- SC-05 監工審核與退回
- SC-06 試算請款與文件產出
- SC-07 關帳與送出請款
- SC-05（吊車）停駐點與噸數彙總看板

### 2.2 角色
- 現場人員、專案經理、監工/主管、請款人員

## 3. 流程（Process）
- **As-Is**：回報與拍照綁定，審核後產出文件
- **To-Be**：回報與拍照拆線，審核可退回，文件產出可篩選且具追溯
- 詳細流程圖請見：`02_Analysis/第二階段_流程圖.md`

## 4. 需求展開（Requirement Decomposition）
### 4.1 功能需求
- **回報/拍照雙軌**（need_report / need_photo）
- **退回與補件**（Returned，可重新回報）
- **照片歷史版本**（Current / History）
- **文件產出條件過濾**（工序欄位＋狀態）
- **設備完工回報**（通知可請款，可重開修改）

### 4.2 資料與欄位
- need_report / need_photo
- photo_status（Pending / Completed / NotRequired）
- crane_tonnage / parking_zone_group

### 4.3 驗證與驗收
- need_photo=否 → 不得阻擋回報
- need_photo=是 → 回報可先完成，photo_status=Pending
- PhotoPending 可核定/請款（與拍照流程分線）
- 退回後可重新回報（回報可再開）
- 文件輸出需記錄篩選條件與產出時間（快照）

## 5. 決策點（Design Decision）
- PhotoPending **可**核定/請款（與拍照流程分線）
- 關帳後是否允許更正（已確認：不可更正）
- 文件產出為快照與 metadata 紀錄（已確認）
- 退回後可重新回報（已確認）
- 吊車欄位只讀（建議）
